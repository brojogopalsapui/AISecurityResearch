CPU–GPU–NPU Partitioning

Modern edge devices and AI PCs increasingly divide work across CPU, GPU, and NPU blocks. This is not only a performance choice; it also defines where data lives, which processor sees which intermediate state, where firmware or driver control is concentrated, and how trust boundaries are distributed across the platform.

Overview

What this topic covers

CPUs handle control-heavy orchestration, preprocessing, and general software tasks. GPUs often absorb throughput-oriented vector and graphics-adjacent workloads. NPUs aim to run AI workloads efficiently under tighter power and thermal envelopes, especially for sustained local inference. The exact split depends on latency targets, model type, software stack maturity, and device policy.

For security analysis, partitioning changes the observation points. A model that appears to run locally may actually move tensors between multiple subsystems. Inputs, outputs, metadata, or intermediate states can cross drivers, memory domains, and execution engines. Understanding who handles what is therefore essential for realistic threat models on client and edge platforms.

Why it matters

Research significance

  • Heterogeneous execution changes where sensitive data resides and which processor can observe it.
  • Partitioning decisions affect power, thermal behavior, latency, and sustained local inference characteristics.
  • Drivers, runtimes, and firmware across CPU, GPU, and NPU boundaries become part of the security surface.
  • Edge and AI PC deployments require trust models that match how work is actually divided on-device.

Discussion

Key technical questions

Heterogeneous systems are valuable because they are flexible, but that flexibility also complicates security interpretation.

Threat model

Can the attacker observe CPU-side orchestration, GPU-side execution, NPU telemetry, shared memory, or inter-processor traffic? The answer changes the realistic attack surface.

System constraints

Client and edge systems are constrained by battery, thermals, sustained performance, and software availability, so work partitioning is often dynamic rather than fixed.

Open direction

A strong open direction is to create methods for tracing sensitive AI data across heterogeneous subsystems without relying on vendor-specific tools that are hard to generalize.
